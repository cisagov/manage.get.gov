# AI Usage Policy
## Purpose
This policy defines acceptable use of AI tools when contributing to this repository. The goal is to allow productivity gains from AI-assisted development while protecting security, privacy, code quality, and contributor accountability.
This repository supports a government or sensitive-data system. Security mistakes have real-world consequences. When uncertain, stop and ask before using AI-generated output or sharing repository context with an AI tool.
## Scope
This policy applies to all repo contributors, including internal contributors, contractors, external collaborators, and anyone submitting pull requests.
This policy applies to AI-assisted work involving:
- Code generation
- Code review
- Refactoring
- Test generation
- Documentation
- Debugging
- Architecture suggestions
- Security analysis
- PR descriptions or summaries
## Approved AI Tool Categories
Contributors may use the following AI tools, subject to this policy:
- GitHub Copilot
- OpenAI Codex
- Claude Code
- Cursor
- ChatGPT web app
- Claude web app
Use of other AI tools should be reviewed before use if they require uploading repository content, logs, screenshots, configuration files, or implementation details.
## Contributor Accountability
AI tools may assist contributors, but they do not replace human engineering judgment.
The human contributor is fully responsible for:
- Understanding every submitted change
- Verifying correctness
- Verifying security impact
- Running appropriate tests
- Ensuring generated code follows project conventions
- Ensuring no sensitive data was shared with or generated by AI tools
- Ensuring no AI output introduces licensing, security, or privacy risk
Do not submit AI-generated code that you cannot explain, test, or maintain.
## Data Handling Rules
### Production Data
Production data must never be shared with AI tools.
This includes, but is not limited to:
- Production database records
- Production logs
- Production user data
- Domain requester data
- Domain manager data
- Email addresses
- Phone numbers
- Mailing addresses
- SSNs
- National IDs
- Any other personally identifiable information, also known as PII
- Registry-sensitive data
- DNS-sensitive data
- Cloud.gov environment output
- Vendor account details
### PII
No PII is allowed in AI prompts, screenshots, pasted logs, pasted stack traces, uploaded files, generated examples, or AI-assisted debugging sessions.
Always remove or replace PII before using AI tools.
Use safe placeholders such as:
```text
user@example.test
example.gov
REDACTED_TOKEN
REDACTED_EMAIL
REDACTED_DOMAIN
REDACTED_ACCOUNT_ID

Secrets and Credentials

Never share secrets with AI tools.

This includes:

* API keys
* Access tokens
* Refresh tokens
* Session cookies
* Private keys
* Certificates
* Cloud.gov credentials
* Login.gov credentials
* AWS credentials
* DNS vendor credentials
* Registry/EPP credentials
* .env files
* Environment variable dumps
* CI/CD secrets
* OAuth/OIDC client secrets

Do not paste full configuration files if they contain secrets or sensitive deployment details. Redact first.

Acceptable AI Use

AI tools may be used for:

* Explaining existing public repository code
* Drafting tests for contributor review
* Suggesting refactors that preserve behavior
* Improving documentation
* Summarizing non-sensitive errors after redaction
* Creating PR descriptions or release notes from reviewed changes
* Generating boilerplate that follows existing project patterns
* Identifying potential edge cases
* Reviewing code for readability or maintainability
* Suggesting security questions for human review

All AI-generated output must be reviewed and validated by the contributor before being committed.

Prohibited AI Use

Contributors must not use AI tools to:

* Process production data
* Process PII
* Process unredacted logs or stack traces
* Process .env files
* Process certificates, private keys, or secrets
* Generate or modify code that bypasses authentication
* Generate or modify code that bypasses authorization
* Expand API token scopes beyond the minimum required
* Disable or weaken security controls
* Disable or weaken audit logging
* Disable or weaken CSRF, CSP, HSTS, secure cookies, or authentication middleware
* Write SQL using string interpolation
* Introduce shell=True in subprocess calls
* Use eval() or exec() with user-supplied input
* Create tests that depend on live production services
* Produce changes the contributor does not understand
* Submit AI-generated code without human review

Restricted Areas Requiring Extra Care

AI tools may assist with these areas, but contributors must apply heightened scrutiny and ensure normal human review covers the risk.

Restricted areas include:

* Login.gov / OIDC authentication
* Authorization and permission checks
* Admin, analyst, requester, portfolio, and domain-manager access control
* CSRF configuration
* CSP configuration
* Security middleware
* Audit logging
* DNS hosting integrations
* Cloudflare or DNS vendor integrations
* EPP registry integrations
* AWS SES or S3 integrations
* Database migrations
* Data migrations
* PII handling
* Logging behavior
* Environment files
* Certificates
* Secrets
* API keys
* CI/CD configuration
* Cloud.gov deployment configuration

For these areas, contributors should prefer small, focused pull requests with clear tests and explicit review notes.

Prompting Guidelines

When using AI tools, provide only the minimum context needed.

Prefer safe prompts such as:

Given this redacted Django view, suggest test cases for permission handling.
Review this redacted function for readability and edge cases. Do not change authentication behavior.
Generate a pytest/Django test outline for this behavior using placeholders only.

Avoid prompts such as:

Here are production logs. Find the bug.
Here is my .env file. Configure the app.
Here is a real user record. Write a migration.
Bypass this permission check so the test passes.

Code Quality Expectations

AI-assisted code must meet the same standards as human-written code.

Contributors must ensure AI-assisted changes:

* Follow existing repository conventions
* Are readable and maintainable
* Are appropriately scoped
* Include tests where behavior changes
* Do not introduce unrelated refactors
* Do not duplicate existing logic unnecessarily
* Do not bypass established service boundaries
* Do not weaken security posture
* Do not introduce unapproved dependencies
* Do not create unreachable or unused code
* Do not add comments that merely restate obvious code

Testing Expectations

Before submitting AI-assisted changes, contributors must run the tests and checks appropriate to the change.

At minimum:

* Run targeted tests for the touched area.
* Run broader test suites when changing shared models, services, settings, middleware, or permissions.
* Run migration checks when models change.
* Run frontend or accessibility checks when templates, assets, or user-facing flows change.
* Run security checks when security-sensitive code changes.

AI-generated tests are not sufficient unless the contributor has reviewed them for meaningful assertions and realistic behavior.

Do not accept shallow tests that only verify implementation details or duplicate the generated code’s assumptions.

AI-Generated Security Suggestions

AI-generated security suggestions must be treated as recommendations, not authority.

Before applying an AI-generated security change:

* Confirm the behavior against existing project patterns.
* Check whether the change affects authentication or authorization.
* Check whether the change affects sensitive data handling.
* Check whether the change affects deployment or production behavior.
* Add tests that prove the intended security behavior.
* Request human review through the normal PR process.

When uncertain, do not merge until the security impact is understood.

Pull Request Expectations

All pull requests must follow the normal human review process.

Contributors should include the standard PR attestation that they:

* Reviewed the submitted change
* Understand the submitted change
* Verified that no secrets or PII were included
* Ran appropriate tests or documented why tests were not run
* Accept responsibility for the change, regardless of whether AI tools were used

Recommended PR checkbox:

- [ ] I reviewed this change and understand its behavior.
- [ ] I verified that this change does not include secrets, credentials, or PII.
- [ ] I ran the relevant tests/checks or documented why they were not run.
- [ ] I accept responsibility for this change, including any AI-assisted portions.

AI use does not need to be disclosed in commit messages or code comments.

In the PR description, disclose AI assistance when it materially influenced:

* Architecture
* Security-sensitive logic
* Authentication or authorization behavior
* Database migrations or data migrations
* Large refactors
* Generated tests that define expected behavior

Suggested optional PR note:

AI assistance was used to draft or review portions of this change. I reviewed the output, verified the behavior, and take responsibility for the final implementation.

Documentation Use

AI tools may be used to draft or edit documentation, provided no sensitive information is included.

Do not use AI tools to process or summarize:

* Non-public incident details
* Production data
* Research participant information
* Procurement-sensitive information
* Compliance documents containing sensitive infrastructure details
* Internal-only operational details unless explicitly approved

Dependency and Licensing Caution

Do not accept AI suggestions to add new packages, libraries, services, or licenses without human review.

Before adding a dependency, verify:

* The dependency is necessary
* The license is acceptable
* The package is actively maintained
* The package does not introduce avoidable security risk
* The dependency is compatible with the repository’s deployment environment

Incident Handling

If sensitive data, secrets, credentials, certificates, or PII are accidentally shared with an AI tool:

1. Stop work immediately.
2. Notify the appropriate project/security contact through the normal internal process.
3. Rotate affected secrets or credentials if applicable.
4. Remove exposed sensitive data from prompts, local files, branches, commits, or PRs where possible.
5. Document the exposure according to project process.

Do not attempt to quietly fix or hide accidental exposure.

Summary

AI tools are allowed as engineering assistants, not autonomous contributors. Contributors must protect sensitive data, preserve security controls, validate generated output, and remain accountable for every submitted change.
